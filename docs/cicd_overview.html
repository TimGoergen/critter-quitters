CI/CD Pipeline — Critter Quitters Pest Control

Build & Deploy overview · GitHub Actions · Last updated May 2026 (ADB deploy added)

1. The Platform: GitHub Actions

The pipeline runs on GitHub Actions, which is GitHub's built-in CI/CD system. If you think of GitHub as a combination of Azure Repos and Azure DevOps, then GitHub Actions is the Pipelines half — it reads a YAML definition file, provisions temporary virtual machines, and executes your build steps on them.

Azure DevOps analogy Think of GitHub Actions as Azure Pipelines with Microsoft-hosted agents, where the pipeline definition lives in a .yml file inside the repository itself. There is no separate pipeline editor UI — everything is code in the repo.

The pipeline definition file is: .github/workflows/build.yml — a path that GitHub Actions looks for automatically. Every repository can have multiple workflow files; this project has one.

2. What Triggers the Pipeline

Automatic trigger — push to main

The pipeline runs automatically whenever a commit is pushed (or a pull request is merged) to the main branch. This is equivalent to a Continuous Integration trigger in Azure DevOps set to watch a specific branch.

No other branches trigger the pipeline. Feature branches can be pushed freely without causing a build.

Manual trigger

The pipeline can also be triggered on demand, without a code change. In GitHub, this is called a workflow dispatch.

To trigger it manually:

  1. Open the repository on GitHub.com.
  2. Click the Actions tab (top navigation bar).
  3. Select Build & Deploy from the workflow list on the left.
  4. Click the Run workflow button (top right of the run list).
  5. Select main as the branch and click Run workflow.
Azure DevOps analogy This is the equivalent of clicking Queue on a pipeline in Azure DevOps. In Octopus terms, it is like triggering a deployment manually from the dashboard.

3. Where It Runs

Each job in the pipeline runs on a GitHub-hosted runner — a temporary virtual machine that GitHub provisions, runs your job on, and then destroys. You do not own or manage these machines. They are the equivalent of Microsoft-hosted agents in Azure DevOps.

Job 1 runner
ubuntu-latest — a Linux VM (Ubuntu 24.x). Used for Godot headless export of both Windows and Android targets, because Linux can cross-compile to both platforms.
Job 2 runner
windows-latest — a Windows Server VM. Required because Inno Setup (the Windows installer builder) only runs on Windows.
Job 3 runner
ubuntu-latest — Linux again, used only to publish the final GitHub release. No compilation happens here.
Job 4 runner
ubuntu-latest — Linux, used to upload the signed APK to Firebase App Distribution. Skipped automatically when the FIREBASE_APP_ID secret is not configured.

Runners have a generous set of software pre-installed (build tools, Java, the Android SDK, PowerShell, Python, etc.), which is why the pipeline does not need to install most of its dependencies from scratch on every run.

Billing note Public repositories on GitHub get unlimited Actions minutes for free. Private repositories have a monthly free allowance (2,000 minutes as of this writing); Linux minutes are cheaper than Windows minutes. This project's pipeline uses roughly 5–8 minutes of Linux time and 1–2 minutes of Windows time per run.

4. Pipeline Structure

GitHub Actions uses the word job where Azure DevOps uses the word stage. Each job runs on its own fresh runner in parallel with other jobs, unless you declare a dependency. This pipeline has three jobs that run in sequence:

Job 1
Export
ubuntu-latest
  • Download Godot (cached)
  • Decode Android keystore
  • Inject signing password
  • Configure Godot settings
  • Export Windows .exe
  • Export & sign Android .apk
  • Upload both as artifacts
Job 2
Windows Installer
windows-latest
  • Download Windows .exe artifact
  • Run Inno Setup compiler
  • Upload installer as artifact
Job 3
Publish Release
ubuntu-latest
  • Download all artifacts
  • Create GitHub pre-release
  • Attach .apk + setup .exe
Job 4
Firebase Deploy
ubuntu-latest
  • Download APK artifact
  • Upload to Firebase App Distribution
  • Skipped if secret absent

Job 2 cannot start until Job 1 finishes successfully. Job 3 cannot start until both Job 1 and Job 2 finish successfully. Job 4 runs independently after Job 1 and is skipped when the FIREBASE_APP_ID secret has not been configured. If any required job fails, downstream jobs are skipped and the overall pipeline is marked failed.

Azure DevOps / Octopus analogy This maps directly to a three-stage Azure Pipeline with sequential stage dependencies, where Job 3 is analogous to an Octopus Deploy step that publishes the release package and makes it available for installation.

5. What Each Job Does, Step by Step

Job 1 — Export (Windows + Android)

StepWhat it does
Checkout Clones the repository onto the runner. Equivalent to a Get Sources task in Azure DevOps.
Compute short SHA Takes the first 7 characters of the commit hash (e.g., c2ad952) and stores it as a pipeline variable for use in filenames and the release tag.
Set up Java 17 Installs Temurin JDK 17. Required by apksigner, the Android signing tool.
Restore Godot from cache Checks whether the Godot binary and export templates have already been downloaded on a previous run. If the cache exists, the next two steps are skipped. The cache key is godot-4.5.1-stable-v5 — incrementing the version number or the suffix busts the cache.
Download Godot binary Downloads the Godot 4.5.1 Linux headless binary from the official Godot GitHub releases page. This is the "server" build of Godot — it has no display output and is designed specifically for CI/CD. (Skipped on cache hit.)
Download export templates Downloads the matching export template package. Export templates are the pre-compiled platform runtimes that Godot packages your game into. Without them, Godot cannot produce a Windows or Android build. (Skipped on cache hit.)
Decode Android keystore Retrieves the Android release keystore from the ANDROID_KEYSTORE_BASE64 secret, decodes it from base64, and writes it to build/android_release.keystore on the runner's local disk. The keystore file itself is never stored in the repository.
Inject keystore password Reads the ANDROID_KEYSTORE_PASS secret and writes it into the game/export_presets.cfg file in the placeholder field that was left intentionally blank in the repository. This keeps the password out of source control.
Verify keystore and SDK Sanity check — confirms the password was injected and the keystore file exists before proceeding. Fails the build immediately if either is missing.
Configure Godot editor settings Writes a Godot editor settings file to the runner's home directory, telling Godot where to find the Android SDK and Java SDK. These paths are fixed on GitHub's Ubuntu runners.
Export — Windows Desktop Runs Godot in headless mode with the --export-release flag. Godot reads the project files, packages all game assets, and produces a standalone .exe for Windows. Linux can cross-compile for Windows because the export uses pre-built Windows templates.
Export — Android APK Runs Godot again to produce an Android .apk. Godot's built-in APK signing is bypassed due to a known bug in Godot 4.5. Instead, the APK is signed immediately afterward using apksigner from the Android build-tools, then the unsigned intermediate file is deleted.
Stamp version into filenames Renames the output files to include the short commit SHA: critter_quitters_c2ad952_win.exe and critter_quitters_c2ad952.apk.
Upload artifacts Stores both output files as pipeline artifacts — temporary storage attached to this pipeline run. Artifacts are the hand-off mechanism between jobs running on different machines. Equivalent to publishing an artifact in Azure DevOps.

Job 2 — Windows Installer

StepWhat it does
Checkout Clones the repository. Needed to access the Inno Setup script (build/setup.iss).
Download Windows exe artifact Retrieves the .exe produced by Job 1 and places it in dist\ on this new Windows runner.
Build installer with Inno Setup Runs the Inno Setup compiler (ISCC.exe), which is pre-installed on GitHub Windows runners, against the build/setup.iss script. This produces a standard Windows installer: critter_quitters_setup_c2ad952.exe. The version string and source file path are passed in as compiler defines.
Upload installer artifact Stores the installer .exe as a pipeline artifact for Job 3 to collect.

Job 3 — Publish Release

StepWhat it does
Download all artifacts Collects both artifacts from Jobs 1 and 2 onto this runner's disk.
Create GitHub pre-release Creates a GitHub Release — a named, versioned snapshot in the repository — tagged build-c2ad952 (using the short SHA). Attaches the Windows installer and the signed Android APK as downloadable assets. The release is marked as a pre-release, which flags it as not yet promoted to production in the GitHub UI.

Job 4 — Firebase Deploy (optional — skipped until secrets configured)

StepWhat it does
Download APK artifact Retrieves the signed .apk produced by Job 1.
Resolve APK path Finds the exact filename of the APK (which includes the SHA) so the Firebase action receives a concrete file path rather than a glob pattern.
Upload to Firebase App Distribution Sends the APK to Firebase App Distribution using the wzieba/Firebase-Distribution-Github-Action action. Firebase notifies registered testers (the testers group) that a new build is available. Testers receive a push notification on their device to install the update. Requires the FIREBASE_APP_ID and FIREBASE_SERVICE_ACCOUNT_KEY secrets to be configured.

6. Secrets and Configuration

Sensitive values are stored as GitHub Actions Secrets, which are equivalent to secret pipeline variables in Azure DevOps. They are encrypted at rest, never echoed in logs, and are only injected into the pipeline at runtime.

Where to find them GitHub repository → Settings → Secrets and variables → Actions. This is equivalent to Azure DevOps pipeline Library variable groups marked as secret.
Secret nameWhat it containsHow it is used
ANDROID_KEYSTORE_BASE64 The Android release keystore file, encoded as base64 text. Decoded back to a binary .jks/.keystore file on the runner for signing.
ANDROID_KEYSTORE_PASS The password protecting the keystore and its key. Injected into the Godot export presets at runtime; also passed to apksigner for signing.
FIREBASE_APP_ID The Firebase Android App ID for this project (e.g. 1:123456789:android:abcdef). Found in Firebase console → Project Settings → Your apps. Used by Job 4. When absent, Job 4 is skipped entirely — the rest of the pipeline is unaffected.
FIREBASE_SERVICE_ACCOUNT_KEY Full JSON content of a GCP service account key with the Firebase App Distribution Admin role. Authenticates the Firebase upload in Job 4. Required alongside FIREBASE_APP_ID.

There is also one pipeline-level configuration value stored directly in the workflow file (not a secret, since it is not sensitive):

GODOT_VERSION
Set to 4.5.1 at the top of build.yml. To upgrade Godot, change this value and clear the Actions cache (repository → Actions → Caches) so the new version is downloaded.

7. Dependency Caching

The Godot binary (~100 MB) and export templates (~700 MB) take several minutes to download. The pipeline caches them between runs so that subsequent builds skip the download entirely.

The cache is keyed on the string godot-4.5.1-stable-v5. If the cache becomes corrupt or you upgrade Godot, either change the version number or increment the v5 suffix. Cached items are stored by GitHub and automatically evicted after 7 days of non-use.

Azure DevOps analogy This is equivalent to the Cache task in Azure Pipelines, which stores and restores directories between pipeline runs using a cache key.

8. Where the Outputs Go

GitHub Release (permanent)

The primary output. Every successful run creates a new GitHub Release tagged build-{sha}, visible at:
Repository → Releases

Each release contains two downloadable files:

critter_quitters_setup_{sha}.exe — Windows installer
critter_quitters_{sha}.apk — signed Android APK

Releases are marked as pre-release. They persist indefinitely until manually deleted.

Pipeline Artifacts (temporary)

Intermediate files produced during the run and used to pass outputs between jobs. Three artifacts are created per run:

windows-exe — raw .exe from Godot
android-apk — signed .apk
windows-installer — Inno Setup installer

Artifacts are available for download from the run's summary page for 90 days, then automatically deleted. They are not the intended distribution mechanism — the Release is.

Octopus Deploy analogy The GitHub Release is the closest equivalent to an Octopus release package. It is versioned, tagged, and contains the actual deployable artifacts. The pipeline artifacts are analogous to the intermediate drop folder in an Azure DevOps build that feeds into the Octopus package feed — useful for debugging, but not the thing you hand to an end user.

9. Installing on Your Pixel (ADB Script)

After CI produces a release, you can install the APK directly onto your Pixel 10 Pro XL using the script scripts/deploy-to-pixel.ps1. The script uses the gh CLI (already configured on this machine) to download the latest APK from GitHub Releases, then installs it via ADB (Android Debug Bridge) over your local WiFi network.

One-time setup

These steps only need to be done once per machine. The ADB tools are already installed at %LOCALAPPDATA%\Android\Sdk\platform-tools\adb.exe.

  1. On your Pixel: Settings → About phone → Build number — tap Build number seven times to unlock Developer options.
  2. Go to Settings → System → Developer options and enable Wireless debugging.
  3. Tap "Pair device with pairing code". The screen shows an IP address, a pairing port, and a 6-digit code.
  4. On your PC, run:
    & "$env:LOCALAPPDATA\Android\Sdk\platform-tools\adb.exe" pair <ip>:<pairing-port>
    Enter the 6-digit code when prompted.
Pairing vs. connecting Pairing is permanent — you only do it once. Connecting is per-session: after a Pixel reboot, the listen port changes and you need to reconnect (but not re-pair).

Per-session reconnect (after each Pixel reboot)

The Wireless Debugging screen shows a second port labeled "IP address & Port" — this is the listen port, separate from the pairing port. Run:

& "$env:LOCALAPPDATA\Android\Sdk\platform-tools\adb.exe" connect <ip>:<port>

You only need this if the Pixel has rebooted since the last session. If ADB is already connected, skip straight to deploying.

Deploying a build

Once ADB is connected, run the deploy script from a PowerShell terminal:

powershell.exe -ExecutionPolicy Bypass -File "D:\Claude\Critter Quitters\scripts\deploy-to-pixel.ps1"

The script will:

  1. Query GitHub Releases for the latest build tag.
  2. Download the signed APK to a temporary folder.
  3. Detect the connected ADB device automatically.
  4. Run adb install -r to install (or update) the app on the Pixel.
  5. Clean up the temporary download.

To install a specific build rather than the latest, pass the release tag:

... deploy-to-pixel.ps1 -Tag build-abc1234

Install from unknown sources The first time you install a sideloaded APK, Android may require you to allow installation from unknown sources. Follow the on-screen prompt on the Pixel. You only need to do this once.

10. Monitoring and Viewing Results

All pipeline runs are visible under the Actions tab of the repository. Each run shows:

  • Overall pass/fail status
  • Which job failed (if any)
  • Full step-by-step logs for each job
  • A list of artifacts produced by the run
  • Total runtime and per-job duration

GitHub can be configured to send email notifications on pipeline failure via Settings → Notifications in your GitHub account preferences.

On failure If a run fails, open the run from the Actions tab, click the failed job, and expand the failed step. The raw log output — including Godot's error messages — is shown inline. There is no separate log storage system to query; everything is in the GitHub Actions UI.
